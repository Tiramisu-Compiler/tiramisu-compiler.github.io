759  /**

761  *
762  * \details Computing the bounds of each computation means computing
763  * the constraints over the iteration domains of each computation in
764  * the function.

766  * In order to deduce bounds, Tiramisu first identifies the final consumers
767  * in the function (i.e., computations that do not have any consumer).
768  * Then, it propagates the bounds over the final consumers to their producers.
769  * The bounds of each consumer are used to deduce the bounds over its producer.
770  *
771  * To take benefit of bound inference, the user can declare computations
772  * without providing constraints on their iteration domains. For example
773  * the user can declare the following computations (the left side is the
774  * iteration domain, while the right side is the expression attached to
775  * each computation)

777  * \code
778  * {A[i] } : 0
779  * {B[i] } : 0
780  * {C[i] } : A[i] + B[i]
781  * {D[i]: 0<=i<N} : 2*C[i]
782  * \endcode

786  * last computations (last consumers), and Tiramisu will propagate
787  * these constraints to all the chain of computations that produce for

790  * only provided for the last consumer "D[i]" and no constraints were
791  * provided for the other computations. Bound inference would deduce
792  * the constraints for the computations A[i], B[i] and C[i].
793  *
794  * Note that bound inference is not possible if you have multiple definitions
795  * of the same computation. For example, if you have multiple definitions
796  * of the same computations, in such a case the user should provide
797  * constraints of the iteration domain of the computation. Example:
798  *
799  * \code

801  * {C[i]: i=0 } : 0

2286  /**
2287  * Set the schedule indicated by \p map.
2288  *
2289  * \p map is a string that represents a mapping from the iteration domain
2290  * to the time-space domain (the ISL format to represent maps is
2291  * documented in http://barvinok.gforge.inria.fr/barvinok.pdf in Sec 1.2.2).
2292  *
2293  * The schedule is a map from the iteration domain to a time space
2294  * domain. The same name of space should be used for both the range
2295  * and the domain of the schedule.
2296  *
2297  * In general, users can set the schedule using high level functions such
2298  * as before(), after(), tile(), compute_at(), vectorize(), split(), ...
2299  * The use of this function is only reserved for advanced users who want
2300  * a low level control of the schedule.
2301  *
2302  * Vectors in the time-space domain have the following form
2303  *
2304  * computation_name[redundancy_ID,static,dynamic,static,dynamic,static,dynamic,static,...]
2305  *
2306  * The first dimension of the vector is used to indicate the redundancy ID
2307  * (the notion of the redundancy ID is explained later).
2308  *
2309  * The following dimensions are interleaved dimensions: static, dynamic, static,
2310  * dynamic, ...
2311  * Dynamic dimensions represent the loop levels, while static dimensions are
2312  * used to order statements within a given block of statements in a given loop
2313  * level.
2314  * For example, the computations c0 and c1 in the following loop nest
2315  *
2316  * \code
2317  * for (i=0; i<N: i++)
2318  * for (j=0; j<N; j++)
2319  * {
2320  * c0;
2321  * c1;
2322  * }
2323  *
2324  * \endcode
2325  *
2326  * have the following representations in the iteration domain
2327  *
2328  * \code
2329  * {c0[i,j]: 0<=i<N and 0<=j<N}
2330  * {c1[i,j]: 0<=i<N and 0<=j<N}
2331  * \endcode
2332  *
2333  * and the following representation in the time-space domain
2334  *
2335  * \code
2336  * {c0[0,0,i,0,j,0]: 0<=i<N and 0<=j<N}
2337  * {c1[0,0,i,0,j,1]: 0<=i<N and 0<=j<N}
2338  * \endcode
2339  *
2340  * The first dimension (dimension 0) in the time-space
2341  * domain (the leftmost dimension) is the redundancy ID
2342  * (in this case it is 0, the meaning of this ID will be explained later).
2343  * The second dimension (starting from the left) is a static dimension,
2344  * the third dimension is a dynamic dimension that
2345  * represents the loop level i, ..., the fifth dimension is a dynamic
2346  * dimension that represents the loop level j and the last dimension
2347  * (dimension 5) is a static dimension and allows the ordering of
2348  * c1 after c0 in the loop nest.
2349  *
2350  * To transform the previous iteration domain to the
2351  * time-space domain, the following schedule should be used:
2352  *
2353  * \code
2354  * {c0[i,j]->c0[0,0,i,0,j,0]: 0<=i<N and 0<=j<N}
2355  * {c1[i,j]->c1[0,0,i,0,j,1]: 0<=i<N and 0<=j<N}
2356  * \endcode
2357 
2358  *
2359  * The first dimension called "redundancy ID" is only meaningful if the
2360  * computation is redundant. i.e., some parts of the computation are
2361  * redundantly computed. Redundant computations are in general used to
2362  * maximize parallelism and data locality on the expense of doing some
2363  * computations redundantly.
2364  * For example, if the two computations c1(i,j) and c2(i,j) both depend
2365  * on the computation c0(i,j), instead of waiting for c0(i,j) to be
2366  * computed and then computing c1(i,j) and c2(i,j) in parallel, the thread
2367  * executing c1(i,j) can compute c0(i,j) by itself and then run c1(i,j).
2368  * The thread that computes c2(i,j) can do the same and compute c0(i,j)
2369  * by itself and then compute c2(i,j). In this case the two threads do not
2370  * need to wait. This is done at the expense of redundant computation since
2371  * c0(i,j) is computed by both threads.
2372  *
2373  * In general redundant computations are useful when tiling stencil
2374  * computations. In the context of stencils such a tiling is called
2375  * "overlapped tiling". Tiles that depend on results computed by other
2376  * tiles that run in parallel can compute the boundaries redundantly which
2377  * allows them to avoid waiting and thus can run in parallel.
2378  *
2379  * In Tiramisu, the user can indicate that a chunk of a computation
2380  * should be computed redundantly. The original computation always has a redundancy
2381  * ID equal to 0 (which means this is the original computation).
2382  * The redundant chunk has an ID that is different from 0 and that is
2383  * used to uniquely identify it.
2384  *
2385  * For example if we want to compute all of c0 three times (that is,
2386  * compute the original computation and compute two redundant computations),
2387  * we can use the following schedules:
2388  *
2389  * The schedule of the original computation: {c0[i,j]->c0[0,0,i,0,j,0]: 0<=i<N and 0<=j<N}
2390  * The schedule of the redundant computation N1: {c0[i,j]->c0[1,0,i,0,j,0]: 0<=i<N and 0<=j<N}
2391  * The schedule of the redundant computation N2: {c0[i,j]->c0[2,0,i,0,j,0]: 0<=i<N and 0<=j<N}
2392  *
2393  * The schedule of c0 in this case would be three maps that map c0[i,j] to
2394  * the three different redundant computations in the time-processor domain:
2395  *
2396  * \code
2397  * {c0[i,j]->c0[0,0,i,0,j,0]: 0<=i<N and 0<=j<N;
2398  * c0[i,j]->c0[1,0,i,0,j,0]: 0<=i<N and 0<=j<N;
2399  * c0[i,j]->c0[2,0,i,0,j,0]: 0<=i<N and 0<=j<N}
2400  * \endcode
2401  *
2402  * The function set_schedule() overrides any other schedule set by the high level
2403  * scheduling functions. Currently the user has to choose between using the high
2404  * level scheduling functions or using this low level set_schedule function. The user
2405  * cannot mix the use of the two in the same program because they are not compatible.
2406  */
2407  // @{
2408  void set_schedule(isl_map *map);
2409  void set_schedule(std::string map_str);

3216  * c1;
3217  * }
3218  * \endcode
3219  *
3220  * have the following representations in the iteration domain
3221  *
3222  * \code
3223  * {c0[i,j]: 0<=i<N and 0<=j<N}
3224  * {c1[i,j]: 0<=i<N and 0<=j<N}
3225  * \endcode
3226  *
3227  * and the following representation in the time-space domain
3228  *
3229  * \code
3230  * {c0[0,0,i,0,j,0]: 0<=i<N and 0<=j<N}
3231  * {c1[0,0,i,0,j,1]: 0<=i<N and 0<=j<N}
3232  * \endcode
3233  *
3234  * The first dimension (dimension 0) in the time-space
3235  * domain (the leftmost dimension) is the redundancy ID
3236  * (in this case it is 0, the meaning of this ID will be explained later).
3237  * The second dimension (starting from the left) is a static dimension,
3238  * the third dimension is a dynamic dimension that
3239  * represents the loop level i, ..., the fifth dimension is a dynamic
3240  * dimension that represents the loop level j and the last dimension
3241  * (dimension 5) is a static dimension and allows the ordering of
3242  * c1 after c0 in the loop nest.
3243  *
3244  * To transform the previous iteration domain to the
3245  * time-space domain, the following schedule should be used:
3246  *
3247  * \code
3248  * {c0[i,j]->c0[0,0,i,0,j,0]: 0<=i<N and 0<=j<N}
3249  * {c1[i,j]->c1[0,0,i,0,j,1]: 0<=i<N and 0<=j<N}
3250  * \endcode
3251  *
3252  * The first dimension called "redundancy ID" is only meaningful if the
3253  * computation is redundant. i.e., some parts of the computation are
3254  * redundantly computed. Redundant computations are in general used to
3255  * maximize parallelism and data locality on the expense of doing some
3256  * computations redundantly.
3257  * For example, if the two computations c1(i,j) and c2(i,j) both depend
3258  * on the computation c0(i,j), instead of waiting for c0(i,j) to be
3259  * computed and then computing c1(i,j) and c2(i,j) in parallel, the thread
3260  * executing c1(i,j) can compute c0(i,j) by itself and then run c1(i,j).
3261  * The thread that computes c2(i,j) can do the same and compute c0(i,j)
3262  * by itself and then compute c2(i,j). In this case the two threads do not
3263  * need to wait. This is done at the expense of redundant computation since
3264  * c0(i,j) is computed by both threads.
3265  *
3266  * In general redundant computations are useful when tiling stencil
3267  * computations. In the context of stencils such a tiling is called
3268  * "overlapped tiling". Tiles that depend on results computed by other
3269  * tiles that run in parallel can compute the boundaries redundantly which
3270  * allows them to avoid waiting and thus can run in parallel.
3271  *
3272  * In Tiramisu, the user can indicate that a chunk of a computation
3273  * should be computed redundantly. The original computation always has a redundancy
3274  * ID equal to 0 (which means this is the original computation).
3275  * The redundant chunk has an ID that is different from 0 and that is
3276  * used to uniquely identify it.
3277  *
3278  * For example if we want to compute all of c0 three times (that is,
3279  * compute the original computation and compute two redundant computations),
3280  * we can use the following schedules:
3281  *
3282  * The schedule of the original computation: {c0[i,j]->c0[0,0,i,0,j,0]: 0<=i<N and 0<=j<N}
3283  * The schedule of the redundant computation N1: {c0[i,j]->c0[1,0,i,0,j,0]: 0<=i<N and 0<=j<N}
3284  * The schedule of the redundant computation N2: {c0[i,j]->c0[2,0,i,0,j,0]: 0<=i<N and 0<=j<N}
3285  *
3286  * The schedule of c0 in this case would be three maps that map c0[i,j] to
3287  * the three different redundant computations in the time-processor domain:
3288  *
3289  * \code
3290  * {c0[i,j]->c0[0,0,i,0,j,0]: 0<=i<N and 0<=j<N;
3291  * c0[i,j]->c0[1,0,i,0,j,0]: 0<=i<N and 0<=j<N;
3292  * c0[i,j]->c0[2,0,i,0,j,0]: 0<=i<N and 0<=j<N}
3293  * \endcode
3294  *
3295  * The function set_schedule() overrides any other schedule set by the high level
3296  * scheduling functions. Currently the user has to choose between using the high
3297  * level scheduling functions or using this low level set_schedule function. The user
3298  * cannot mix the use of the two in the same program because they are not compatible.
3299  */
